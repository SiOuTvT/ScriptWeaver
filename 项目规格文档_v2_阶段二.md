# 智能视觉小说编辑器 - 阶段二规格文档：假数据UI骨架

> 前置条件：阶段一（Delta类型定义 + 状态归约函数 + 单元测试）已完成并通过验证。本阶段目标是把界面骨架搭出来，用写死的假数据验证布局、比例、交互动画是否符合预期，暂不接入真实拖拽和文件系统。

---

## 一、本阶段范围边界（重要，防止AI越界多做）

**要做的：**
- 四层布局的组件结构和CSS/Tailwind样式
- 抽屉的三态切换（收起/半展开/钉住）及动画
- 预览与剧本流联动的交叉淡入淡出效果
- 时间轴色块的静态渲染（用假数据展示"连续区间"的视觉效果）
- 侧边栏的折叠/展开交互

**不要做的：**
- 真实的拖拽写入逻辑（素材库拖到舞台/时间轴，先做UI响应即可，不用真的改数据）
- 真实文件系统读写
- Ren'Py导出功能
- 任何AI调用逻辑

## 二、假数据结构（Mock Data）

直接复用阶段一定义的 `LineDelta` 类型，手写一份包含 8-10 行的假剧本数据，需要覆盖以下场景，用于验证UI在各种情况下的表现：

```typescript
// mockScript.ts 应包含的测试场景：
// 1. 第1行：背景+角色A登场（建立初始状态）
// 2. 第2-3行：角色A不做任何指令 → 验证UI是否正确显示"继承"（时间轴色块应连续不断）
// 3. 第4行：角色A切换表情，位置不变 → 验证局部更新
// 4. 第5行：角色B登场，角色A保持 → 验证多角色轨道同时渲染
// 5. 第6行：BGM开始播放 → 验证音频轨道色块起点
// 6. 第7-8行：BGM未声明 → 验证音频轨道色块应跨行延续，不中断
// 7. 第9行：BGM显式停止(__CLEAR__) → 验证色块在此行截断
// 8. 第10行：切换背景+全部角色退场 → 验证转场
```

这份假数据文件是本阶段最先要写的东西，后续所有组件都基于它渲染。

## 三、组件结构与交互细节

### 3.1 最左侧边栏
- 折叠态：仅图标（章节树/角色库/设置/AI入口四个图标即可，先不用做真实跳转，点击console.log占位）
- 展开态：点击图标后侧边栏宽度动画展开，显示对应假数据列表（比如假的章节树：第一章/第二章）

### 3.2 素材库
- Tab切换：背景/立绘/音频三个Tab，用假图片占位（可用纯色块+文字标注代替真实素材图）
- 素材卡片支持鼠标悬停高亮，暂不需要真实可拖拽（阶段三再做）

### 3.3 舞台预览（核心）
- 根据"当前选中行"的 mock LineDelta，渲染对应的背景色块+角色位置色块（用不同颜色矩形代表不同角色，标注角色名即可，先不用真实立绘图片）
- 角色矩形的位置严格按照 `position_slot` 对应的归一化坐标定位，验证槽位系统的定位逻辑本身没问题
- 切换选中行时，触发 200ms 交叉淡入淡出过渡动画

### 3.4 剧本流抽屉
- 三态实现：
  - 收起：完全隐藏，仅保留一个呼出的边缘把手
  - 半展开：从右侧滑入，宽度30%，毛玻璃背景（`backdrop-filter: blur`），预览区域同步做轻微变暗处理
  - 钉住：转为常驻布局，不再有滑入滑出动画，宽度固定
- 快捷键：`Tab` 呼出/收起（用React的键盘事件监听实现）
- 列表渲染假数据的 dialogue 文本，当前选中行高亮显示
- 点击任意行 → 更新"当前选中行"全局状态 → 触发预览联动

### 3.5 底部时间轴
- 横向多轨道：背景轨、角色轨(每角色一条，根据假数据中出现的角色ID动态生成)、BGM轨、环境音轨、SE轨、语音轨
- 每条轨道用色块表示区间：色块的起止列 = 该状态在假数据中连续未变化的行范围
- **这是本阶段验证的重点**：色块必须能正确反映"继承"关系——如果几行数据角色状态没变，视觉上必须是同一个连续色块，而不是断开的多个色块
- 点击色块 → 联动跳转到该区间起始行，触发抽屉展开+预览联动

## 四、验收标准（做完让AI自查这几点）

1. 时间轴上，连续继承的行是否正确渲染为**同一个不间断的色块**（这是最容易出bug的地方，务必重点检查）
2. 点击时间轴任意色块，是否正确联动到对应的剧本行和预览画面
3. 抽屉三态切换动画是否流畅，钉住模式下是否不再有滑入滑出的跳动感
4. 角色在预览中的位置，是否严格按槽位坐标定位，不同行如果引用同一个槽位ID，位置必须逐像素一致（用浏览器开发者工具量一下，不能靠肉眼估）
5. 选中行切换时，预览的交叉淡入淡出是否生效，没有生硬跳变
