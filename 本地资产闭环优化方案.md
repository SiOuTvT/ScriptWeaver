# 本地资产闭环优化方案（C 方向 · Spec 驱动论证）

> 目标：在不破坏现有 Zustand 状态机的前提下，去除 Base64 内存毒瘤，建立 `sw-asset://` 自定义安全协议直读本地硬盘，并实现项目根目录持久化 + 素材库增量扫描/监听。
> 当前证据全部来自工作区真实代码（Electron `^31.7.7`，Zustand `^4.5.4`，用户 OS：win32）。

---

## 0. 范围与边界

| 项 | 本次范围 | 不在范围 |
|---|---|---|
| 资产存储形态 | 移除 Base64，改协议直读 | A/B 方向的 Rpy 导出、AI 打点 |
| 协议 | `sw-asset://` 主进程注册 | 网络远程资源、CDN |
| 状态机 | 复用 `addAsset/deleteAsset/setProjectRoot` 等既有 action | 重写 `reducer.ts` / `positionSlots.ts` 内核 |
| 持久化 | `projectRoot` + 文件夹监听 | 整库 `zustand/persist` 化（另立议题） |

**硬约束**：现有 action 签名不变、delta/角色内核不变、`AssetManager` 与 `StagePreview` 的引用关系不变。所有改动收敛在「导入流程 + 渲染 src 解析 + 新增同步模块」三处，通过既有 store action 接入。

## 1. 现状诊断：Base64 内存毒瘤的实锤

### 1.1 注入点 —— 导入即把整文件塞进内存
`electron/main.ts` 的 `dialog:pickAssetFiles` 在把文件复制到临时目录后，对**图片**直接读成 base64 返回：

```251:251:electron/main.ts
          dataUrl = `data:${mime};base64,${buf.toString('base64')}`
```

音频在导入阶段未转 base64，但会在**回读点**补转（见 1.2）。

### 1.2 回读点 —— 打开项目后全量灌 Base64
`electron/main.ts` 的 `fs:readAssetFile` 对图片和音频都返回 base64：

```306:306:electron/main.ts
      return { success: true, dataUrl: `data:${mime};base64,${buf.toString('base64')}` }
```
```315:315:electron/main.ts
      return { success: true, dataUrl: `data:${mime};base64,${buf.toString('base64')}` }
```

渲染端在打开项目时**逐个 IPC 回读并写回 store**（`AppLayout.refreshAssetDataUrls`）：

```82:93:src/components/layout/AppLayout.tsx
  const refreshed = await Promise.all(
    assets.map(async (asset) => {
      if (asset.dataUrl) return asset
      if (!asset.relativePath) { ... return asset }
      const result = await api.readAssetFile(asset.relativePath, projectRoot)
      if (result.success && result.dataUrl) {
        return { ...asset, dataUrl: result.dataUrl }
      }
```

### 1.3 常驻点 —— `AssetItem.dataUrl` 永久驻留 store
```40:41:src/core/types.ts
  /** 图片/音频的 base64 data URL，用于渲染时直接显示，避免 file:// 安全限制 */
  dataUrl?: string
```
每个素材对象都挂一份完整二进制字符串，常驻 Zustand `assets` 数组。

### 1.4 消费点（共 5 处，全部 `src={asset.dataUrl}`）
- `AssetManager.tsx:316 / :323`（列表缩略图）
- `SceneNavPanel.tsx:159`（音频试听）
- `StagePreview.tsx:124 / :140 / :149`（背景/立绘）
- `audioManager.ts:27 / :36`（预览播放）
- `AppLayout.tsx:85-93`（如上，打开项目回灌）

### 1.5 代价量化（为什么必须干掉）
1. **内存爆炸**：1 张 500KB sprite → base64 膨胀 ~33% ≈ 667KB；50 段 3MB 音频 → base64 占 ~200MB，且常驻不释放。
2. **Undo/Redo 雪崩**：`appStore` 在 `setAssets` 等路径用 `structuredClone(s.assets)`（见 `appStore.ts:505/521/542`）。每次历史节点 clone **整个含 base64 的 assets 数组**，是 O(字节数) 成本，素材多时卡顿明显。
3. **解码缓冲重复**：大量 `<img src={dataUrl}>` 同时解码，浏览器为每个 dataUrl 分配独立解码缓冲，内存峰值叠加。
4. **序列化膨胀**：`serializeProject` 会把 assets 写进 `.swproj`，base64 让 JSON 体积爆炸、读写变慢。

### 1.6 旁证：目录命名不一致（方案需顺手统一）
- 导入 `pickAssetFiles` 写 `assets/sprites` 与 `assets/audio`（`electron/main.ts:225,238`）。
- 保存 `saveProject` 却期望 `assets/backgrounds`、`assets/sprites`、`assets/audio`（`electron/main.ts:124-126`）。
两类命名错位，是后续「增量扫描」必须统一规范的信号。

## 2. 总体架构：从「内联 Base64」到「协议直读」

```
                ┌──────────── 主进程 (Main) ────────────┐
用户 MyGame/  →  projectRoot 持久化 (draftStorage)
                assets/images/{background,sprite}/*
                assets/audio/*                          │
                fs.watch(recursive) ──IPC──┐            │
                protocol.handle('sw-asset') │            │
                          ▲                 ▼            │
                sw-asset://<rootKey>/<rel>  asset:changed│
                          │                 │           │
┌──────────── 渲染进程 (Renderer) ─────────┴───────────┘│
│  resolveAssetSrc(asset) → sw-asset://...  │           │
│  <img>/<audio> src 直接直读硬盘           │           │
│  assetSync 监听 IPC → addAsset/deleteAsset│◄──────────┘
│  (复用既有 store action，内核不动)        │
└──────────────────────────────────────────┘
```

核心转变：**store 只存「元数据 + relativePath」，二进制永远不进内存；渲染层通过协议 URL 向主进程「按需、流式、零拷贝」要文件。**

## 3. 模块一：干掉 Base64 内存毒瘤

### 3.1 `AssetItem` 改造
删除 `dataUrl` 字段（主路径彻底不用）；Web 降级模式改用可选 `blobUrl?: string`（`URL.createObjectURL` 结果，比 dataUrl 省内存，且在页面卸载时 `revokeObjectURL` 释放）。

```diff
  /** 音频时长（秒） */
  duration?: number
- /** 图片/音频的 base64 data URL ... */
- dataUrl?: string
+ /** Web 降级模式下的临时对象 URL（Electron 模式恒为空） */
+ blobUrl?: string
  color?: string
```

### 3.2 统一解析函数 `resolveAssetSrc`
新增 `src/utils/assetSrc.ts`，**所有消费端集中调用**，杜绝散落改动：

```ts
import type { AssetItem } from '@/core/types'

const ROOT_KEY = (() => localStorage.getItem('sw-root-key'))()

export function resolveAssetSrc(asset: AssetItem): string | undefined {
  if (asset.blobUrl) return asset.blobUrl          // Web 降级
  if (!asset.relativePath) return undefined
  // Electron：交给 sw-asset:// 协议流式直读
  return `sw-asset://${ROOT_KEY ?? 'proj'}/${asset.relativePath.replace(/\\/g, '/')}`
}
```

### 3.3 消费端改造清单（仅替换 src，逻辑不动）
| 文件 | 原 | 改 |
|---|---|---|
| `AssetManager.tsx:316/323` | `src={asset.dataUrl}` | `src={resolveAssetSrc(asset)}` |
| `SceneNavPanel.tsx:159` | `src={asset.dataUrl}` | `src={resolveAssetSrc(asset)}` |
| `StagePreview.tsx:124/140/149` | `return asset?.dataUrl` 等 | `return resolveAssetSrc(asset)` |
| `audioManager.ts:27/36` | `asset.dataUrl` / `readAssetFile` | `resolveAssetSrc(asset)`（优先协议直读，失败再 IPC） |
| `AppLayout.tsx:82-93` | 全量回灌 dataUrl | **删除** `refreshAssetDataUrls`，改为调用 `startAssetWatch`（模块三） |

`audioManager` 改为：先 `resolveAssetSrc` 拿到 `sw-asset://` 直读；仅当返回 undefined（Web 模式）才回退 IPC。

## 4. 模块二：Electron 自定义安全协议 `sw-asset://`

### 4.1 注册（必须在 `app.whenReady()` 之前）
```ts
// electron/main.ts —— 在 app.whenReady().then(...) 之前
protocol.registerSchemesAsPrivileged([
  {
    scheme: 'sw-asset',
    privileges: {
      secure: true,        // 享受同源/HTTPS 同等安全策略
      standard: true,      // 被当作标准 scheme，<img>/<audio> src 直接可用
      supportFetchAPI: true,
      stream: true,
      bypassCSP: false,    // 不绕过 CSP，安全性由 CSP 兜底
    },
  },
])
```

### 4.2 Handler 实现（流式零拷贝 + 安全校验）
Electron 31 支持 `protocol.handle`（Fetch 风格，非阻塞，返回 `Response`）。用 `Readable.toWeb` 把文件流直接喂给 `Response`，**大文件不整体进内存**：

```ts
import { Readable } from 'stream'

// 主进程维护 projectRoot ↔ rootKey 映射，避免把绝对路径泄露到渲染进程
const rootMap = new Map<string, string>() // rootKey -> projectRoot

protocol.handle('sw-asset', (request) => {
  try {
    const url = new URL(request.url)
    const [rootKey, ...rest] = url.hostname ? [url.hostname, url.pathname] : url.pathname.slice(1).split('/')
    const projectRoot = rootMap.get(rootKey)
    if (!projectRoot) return new Response('unknown root', { status: 404 })

    const rel = decodeURIComponent(rest.join('/').replace(/^\//, ''))
    const abs = path.resolve(projectRoot, 'assets', rel)
    // 防目录穿越：解析后必须仍在 assets 子树内
    const assetsDir = path.resolve(projectRoot, 'assets')
    if (!abs.startsWith(assetsDir + path.sep) && abs !== assetsDir) {
      return new Response('forbidden', { status: 403 })
    }
    // 扩展名白名单
    const ALLOW = new Set(['.png','.jpg','.jpeg','.webp','.gif','.mp3','.ogg','.wav','.flac'])
    if (!ALLOW.has(path.extname(abs).toLowerCase())) {
      return new Response('type not allowed', { status: 415 })
    }
    if (!fs.existsSync(abs)) return new Response('not found', { status: 404 })

    const mime = mimeFromExt(abs)
    return new Response(Readable.toWeb(fs.createReadStream(abs)), {
      headers: { 'Content-Type': mime, 'Cache-Control': 'no-cache' },
    })
  } catch {
    return new Response('error', { status: 500 })
  }
})
```

**安全要点**：
- 渲染进程只拿到 `sw-asset://<rootKey>/<rel>`，**看不到真实磁盘路径**，规避用户目录结构泄露。
- `path.resolve` + `startsWith(assetsDir)` 双重拦截 `../` 穿越。
- 扩展名白名单 + `bypassCSP:false`，即便被注入也只能在 assets 子树内读白名单类型。

### 4.3 URL 设计与 rootKey 注入
- 打开/保存项目时，主进程生成 `rootKey = uuid()`，`rootMap.set(rootKey, projectDir)`，并通过 `appStore.setProjectRoot` 的扩展字段（或单独 IPC）把 `rootKey` 传给渲染端 → 渲染端存 `localStorage['sw-root-key']`，供 `resolveAssetSrc` 拼 URL。
- `<img src="sw-asset://<rootKey>/sprites/alice_smile.png" />`、`<audio src="sw-asset://<rootKey>/audio/bgm.mp3" />` 即可丝滑直读。

### 4.4 CSP 配合
`index.html` 的 CSP 需放行协议源（当前无显式 CSP，本项为新增保障）：
```
img-src 'self' sw-asset: data:;
media-src 'self' sw-asset: data:;
```

### 4.5 Preload 暴露
`electron/preload.ts` 新增 `registerAssetRoot(rootKey, projectRoot)` 与现有 `electronAPI` 合并（仍走 `contextBridge`，`contextIsolation:true` 不变）。

## 5. 模块三：资产目录持久化 + 增量扫描/监听

### 5.1 `projectRoot` 持久化
现状：`appStore.projectRoot` 在构造时 `null`，打开/保存时由 `setProjectRoot` 写入内存，但**无持久化**，重启即丢（`appStore.ts:152,341`）。

方案：扩展既有 `src/utils/draftStorage.ts`（其 `saveDraft` 已存 `projectRoot`，见 `draftStorage.ts:17,30`），在应用启动时 `loadDraft()` 恢复 `projectRoot`，并在 `setProjectRoot` 时同步写回。无需引入 `zustand/persist`，复用既有 draft 通道，零新增依赖、零状态机改造。

### 5.2 目录规范统一（修复 1.6 错位）
统一为：
```
MyGame/
  game.swproj
  assets/
    images/
      background/   (背景图)
      sprite/       (立绘)
    audio/
      bgm/ ambient/ se/ voice/   (可选细分，扫描时按扩展名归类)
```
`relativePath` 一律以 `assets/...` 起；导入与保存共用同一套落盘逻辑（抽成 `electron/main.ts` 的 `importAssetToProject(absSrc, type)` 私有函数，消除 1.6 的命名错位）。

### 5.3 初次扫描（打开项目）
新增 IPC `fs:scanProjectAssets(projectRoot)`：主进程递归扫描 `assets/images` + `assets/audio`，返回清单 `{ id, name, fileName, relativePath, type, width?, height?, duration? }`。渲染端拿到清单后**直接 `setAssets(list)`**（复用既有 action），不再逐个 IPC 回读 base64。

### 5.4 运行时增量监听（核心增量能力）
```ts
// 主进程
let watcher: fs.FSWatcher | null = null
function startAssetWatch(projectRoot: string) {
  stopAssetWatch()
  const assetsDir = path.join(projectRoot, 'assets')
  watcher = fs.watch(assetsDir, { recursive: true }, (event, filename) => {
    if (!filename) return
    const abs = path.join(assetsDir, filename)
    const type = classifyByExt(abs)
    if (!type) return
    // 防抖后推送增量 diff 给渲染端
    mainWindow?.webContents.send('asset:changed', {
      event, // 'rename' | 'change'
      relativePath: path.relative(assetsDir, abs).replace(/\\/g, '/'),
      type,
    })
  })
}
```
- **win32**（当前目标平台）：`fs.watch(recursive:true)` 原生可靠。
- **macOS/Linux 兼容**：若后续跨平台，引入 `chokidar`（新增一个 devDep）替代原生 watch，API 一致、零逻辑改动。
- **重命名处理**：Windows 的 `rename` 表现为「旧路径 delete + 新路径 add」。采用保守且正确的策略：`delete` 事件 → 渲染端 `deleteAsset(byRelativePath)`；`add` 事件 → 渲染端 `addAsset(...)`。若需更优（保留 assetId 与引用关系），可维护 `inode → assetId` 缓存做匹配，但**首版以 delete+add 保稳定**，引用断链由既有 `deleteAsset` 的 `getRefs` 校验兜底。

### 5.5 渲染端同步（复用既有 action）
新增 `src/services/assetSync.ts`（独立模块，不碰内核）：
```ts
export function bindAssetWatcher() {
  const api = window.electronAPI
  api?.on('asset:changed', (e: { event: string; relativePath: string; type: string }) => {
    const store = useAppStore.getState()
    if (e.event === 'rename' || e.event === 'change') {
      const existing = store.assets.find((a) => a.relativePath === e.relativePath)
      if (existing) {
        // 文件内容变了但相对路径没变：保留 id 与引用，仅刷新元数据
        store.updateAsset(existing.id, { /* 重新取尺寸/时长 */ })
      } else {
        store.addAsset(buildAssetFromDisk(e))  // 复用 addAsset
      }
    } else {
      const existing = store.assets.find((a) => a.relativePath === e.relativePath)
      if (existing) store.deleteAsset(existing.id)  // 复用 deleteAsset（含引用校验）
    }
  })
}
```
**关键**：增量同步**只调用既有 `addAsset/updateAsset/deleteAsset`**，状态机内核（`reducer.ts` / `positionSlots.ts`）完全不被触及 —— 这正是「不破坏 Zustand 状态机」的落点。

## 6. 不破坏 Zustand 状态机的约定（承诺清单）

1. **action 签名不变**：`addAsset / updateAsset / deleteAsset / getAsset / setProjectRoot / setAssets` 全部保留，仅 `setProjectRoot` 内部多写一笔 draft（复用 `draftStorage`）。
2. **内核零改动**：`reducer.ts`、`positionSlots.ts`、delta 数据结构、角色表情引用模型不动。
3. **删除链路不变**：`deleteAsset` 的 `getRefs`（角色表情引用校验）原样生效，增量删除同样受其保护。
4. **`AssetItem` 是结构变更但向后兼容**：旧 `.swproj` 中残留的 `dataUrl` 字段在 `deserializeProject` 时**直接忽略**（不渲染、不回灌），按 `relativePath` 从磁盘解析，旧项目可正常打开。
5. **内存反而更稳**：移除 base64 后，`structuredClone(s.assets)` 的 undo/redo 成本骤降，操作更流畅。

## 7. 兼容性、风险与回退

| 风险 | 缓解 |
|---|---|
| 协议 handler 异常 | 返回 404/500 `Response`，UI 显示占位图（`StagePreview` 已有 `BG_COLORS`/`SPRITE_COLORS` 兜底） |
| 大音频首次解码缓冲 | 流式 `Response` 边下边播，不在内存常驻整文件 |
| Web 模式（无 electronAPI） | `resolveAssetSrc` 走 `blobUrl` 兜底；保持现有降级体验 |
| `protocol.handle` 需 Electron ≥25 | 当前 `^31.7.7` ✓ |
| CSP 误拦协议资源 | 4.4 显式放行 `sw-asset:` |
| 文件夹监听丢失事件（编辑器批量改名） | 5.4 防抖 + `asset:changed` 后渲染端可选触发一次 `fs:scanProjectAssets` 全量校准 |

## 8. 落地执行顺序（与 A/B 方向解耦）

1. **C-1** 注册 `sw-asset://` 协议 + handler + preload 暴露（模块二，基础设施）。
2. **C-2** 抽 `importAssetToProject` 统一目录规范，导入流程停写 base64（模块一注入点）。
3. **C-3** 引入 `resolveAssetSrc`，改造 5 处消费端 + 删除 `refreshAssetDataUrls`（模块一消费点）。
4. **C-4** `projectRoot` 经 `draftStorage` 持久化 + `rootKey` 注入（模块三 5.1/5.3）。
5. **C-5** 初次扫描 `fs:scanProjectAssets` + 运行时 `fs.watch` + 渲染端 `assetSync` 增量同步（模块三 5.3/5.4/5.5）。
6. **C-6** 跑 `npm run build`，零错误零警告；桌面端实测导入/打开/改名/删除联动。

> 以上 6 步全部收敛在「导入流程 + src 解析 + assetSync 新模块 + 协议 handler」，**不触碰 A/B 方向（Rpy 导出、AI 打点）与 Zustand 内核**，可独立合入、独立回退。

## 9. 一句话总结（给决策层）

Base64 是「把整座图书馆复印进会议室」；`sw-asset://` 是「给会议室装一扇直通图书馆的门，按需取书、看毕即还」。本方案用一扇门（协议）+ 一个管理员（增量监听）+ 一张借书卡（rootKey 持久化），在**完全不动状态机内核**的前提下，把内存毒瘤连根拔起，并让素材库具备「打开即同步、改动实时反映」的活体能力。
