# 文档规范（ScriptWeaver 团队 house style）

> 本文件定义"文档怎么写"。所有贡献者写文档前先读一遍。目标：开发者真能读、真能用的文档。

## Divio 四分类（每条文档只服务一种目的）

| 类型 | 目的 | 典型文件 |
|---|---|---|
| **Tutorial**（教程） | 教学、带着从零到跑通 | `GETTING_STARTED.md` |
| **How-to**（操作指南） | 完成任务导向 | 未来的 `how-to/*.md` |
| **Reference**（参考） | 信息导向、可查 | `IPC_AND_EXPORT.md` |
| **Explanation**（解释） | 理解导向、讲 why | `ARCHITECTURE.md` |

> 不要把教程和参考混在一篇里；读者找不到东西，文档就失败了。

## 写作铁律

1. **第二人称 + 现在时 + 主动语态**：写"你运行 `npm run dev`"，不写"开发服务器由用户启动"。
2. **代码样例必须能跑**：每个片段在干净环境实测过再发布；带版本、带前置。
3. **一个概念一节约**：不要把安装、配置、用法堆成一面墙。
4. **零上下文假设**：每篇独立成篇，或显式链接前置上下文。
5. **版本对齐**：文档必须匹配它所描述软件的版本；破坏性变更先出迁移说明，旧文档标记弃用、绝不删除。
6. **README 过 5 秒测试**：访客 5 秒内要知道"这是什么、我为何要在意、怎么开始"。

## 与本仓库工程纪律对齐

文档本身也应体现仓库的强纪律：

- **单一数据源**：文档里"状态/契约"以代码为唯一真相源（如 `RpyBundle` / `AppState` / `.swproj` 结构），不要另写一份会漂移的描述。
- **边界即契约**：讲 IPC / 导出器 / 加载器时，明确写出"调用方保证什么、实现方保证什么"，这是本项目历史严重问题的共性根因。
- **零 `any` 精神**：文档示例中的类型标注要真实、可编译，不要写 `any` 糊弄。

## 格式与链接

- 用相对路径在 `docs/` 内互链（如 `[架构说明](ARCHITECTURE.md)`），不要用绝对路径。
- 标题层级从 `#` 开始，不跳级。
- 代码块标注语言（`bash` / `ts` / `json` / `text`），便于高亮与 lint。
- 表格用于并列对照，长列表用有序/无序列表。

## 质量门禁（CI 建议）

- 文档随代码同 PR 交付；缺文档的 feature PR 视为不完整。
- 引入破坏性变更，PR 描述须含迁移步骤，否则打回。
- 推荐接入 `markdownlint` + 等价 house style 检查，失败即阻断构建。
- 对时间敏感内容（安全、弃用、版本）设定期复核日历。
